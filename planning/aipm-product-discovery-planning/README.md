# AIPM Bootcamp – MVP Planning & Strategic Execution

This work builds upon: product discovery, prototype and user research results.

## 🎯 Goal
Translate everything you discovered so far (personas, prototype, user insights, metrics, datasets) into a **realistic 3.5-week MVP plan** that two people can actually build.

You’ll learn to scope, prioritize, and plan like an **AI Product Manager** — balancing learning value, technical feasibility, and user impact.

---

## 📋 Framework Overview
**Total Time:** ≈ 3 hours individual + 2 hours team synthesis  

1. **MVP Scoping** – Define what to build in 3.5 weeks  
2. **Outcome Goals** – Express what success looks like  
3. **Now-Next-Future Roadmap** – Sequence learning & delivery  
4. **Technical Breakdown** – Check feasibility  
5. **Effort Prioritization** – Fit to team capacity  
6. **User Stories & Tasks** – Plan implementation  

---

## 🧠 Learning Objectives
By the end of this module, you will:
- Identify an MVP that validates your core hypothesis  
- Define measurable outcome goals linked to user value  
- Turn features into buildable deliverables and daily tasks  
- Connect metrics & learning objectives to your roadmap  
- Practice the **AI PM role** of translating strategy into execution

## 📖 Key Terminology for AI PM Planning

| Term | Short Definition / Purpose |
|------|----------------------------|
| **MVP (Minimum Viable Product)** | The smallest build that demonstrates your product’s core value and tests your hypothesis. |
| **Outcome Goal** | A concise statement of user impact + measurable result within a set time frame (“In 3.5 weeks users will…”). |
| **Hypothesis** | The assumption you want to validate (e.g., “If users see AI matches, they’ll trust them more than CV keywords”). |
| **Now-Next-Future Roadmap** | Lightweight time-sequenced view of work (what we build now, what comes after, what’s future vision). |
| **Epic** | A large feature or theme of work (e.g., “Skills Extractor” or “Motivation Matcher”). |
| **Deliverable / Objective** | A buildable component that makes an Epic real (API, UI, dataset, model). |
| **User Story** | Short narrative of user intent → “As a [user], I want to [action] so that [benefit].” |
| **Acceptance Criteria** | Specific conditions that make a story “done” (e.g., form submits, API returns JSON, data saved). |
| **Technical Work Breakdown** | Listing each feature’s components (data, model, backend, frontend, integration, analytics). |
| **Effort Estimate** | Rough size of work → S (1–2 days) / M (3–5 days) / C (6 +). |
| **Prioritization Matrix** | Grid (Value × Effort) that decides what to build now vs later. |
| **Dependency** | Any task that must finish before another can start (e.g., API before UI integration). |
| **Task / Ticket** | Concrete work unit ≤ 1 day, owned by one person or pair. |
| **Learning Goal** | Personal objective for each team member (e.g., learn prompt design or API integration). |
| **Metric** | Quantitative measure used to validate success (e.g., % users who trust recommendations). |
| **Feedback Loop** | Mechanism to collect user data and feed it back into product decisions. |
| **Feasibility Check** | Evaluation of whether your plan fits time, team skills, and available tools. |
